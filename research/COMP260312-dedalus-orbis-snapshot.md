# 6. Legacy UX Traits of ORBIS

| Legacy UX Pattern | What it looks like | Why users complain | Proposed solution |
| --- | --- | --- | --- |
| Information-dense screens | Screens filled with tables and fields | Hard to scan quickly | Break the interface into task-based views with progressive disclosure. Show the most critical patient and workflow data by default, and let users expand secondary details only when needed. |
| Nested windows | Many pop-ups and dialogs | Breaks workflow | Replace modal chains with inline expansion or side panels that keep users anchored in the same workspace. This reduces context switching and makes it easier to review information while taking action. |
| Menu-heavy navigation | Deep hierarchical menus | Slow feature discovery | Flatten navigation around high-frequency clinical tasks rather than system structure. Add global search, favorites, and role-based shortcuts so common actions are reachable in one step. |
| Form-centric UI | Large static forms | Feels administrative | Convert long forms into grouped sections or guided task flows with clear completion states. Reveal only fields relevant to the current patient, role, or workflow stage to reduce cognitive load. |
| Small click targets | Tiny icons and checkboxes | Difficult during fast-paced work | Increase hit areas, spacing, and visual affordance for interactive controls. This improves accuracy in high-pressure environments and makes the UI more usable on touch-enabled devices. |
| Weak visual hierarchy | Important info not emphasized | Hard to prioritize information | Use stronger typography, spacing, color cues, and alert styling to separate critical information from background data. The highest-risk patient signals should be understandable within a few seconds. |
| Limited dashboards | Few overview pages | Hard to grasp patient status quickly | Introduce dashboards for patient, ward, and task overviews with filters and drill-down paths. Users should be able to identify risk, backlog, and next actions before opening detailed records. |
| Fragmented workflows | Tasks spread across modules | Requires many clicks | Redesign common workflows end to end so related actions happen in one connected flow. Preserve patient context between steps to reduce duplicate navigation and re-entry. |
| Desktop-first design | Built for hospital terminals | Poor tablet usability | Apply responsive layouts to the highest-frequency bedside and on-the-go tasks first. The interface should adapt to smaller screens without hiding essential actions or status information. |
| Inconsistent interactions | Modules behave differently | High learning curve | Standardize navigation, editing, confirmation, and feedback patterns across modules. A shared design system makes new screens more predictable and reduces retraining effort. |

# 7. Why the UX Looks This Way

ORBIS design reflects historical HIS constraints:

-   Built for desktop terminals
-   Optimized for data completeness
-   Developed before modern UX practices
-   Designed for experienced hospital staff rather than new users

------------------------------------------------------------------------

# Key Takeaway

ORBIS prioritizes **data completeness, configurability, and operational
reliability**\
over **visual clarity and workflow simplicity**.
