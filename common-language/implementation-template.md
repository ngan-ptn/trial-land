# Prototype Implementation Template

> Use this template to translate a chosen solution approach into a **buildable prototype**.  
> Focus on **user intent, UX logic, and interaction clarity** — not engineering or system design.

---

## 1. Core Concept

**One-liner**  
> Summarize the idea in a single, human sentence.

**Expanded description (2–3 sentences)**  
> Explain what the experience is, for whom, and the core value it delivers.  
> Avoid feature lists — focus on *experience shift*.

---

## 2. UX Rationale

**User problem being addressed**
- What user frustration, anxiety, or inefficiency does this solve?
- Why existing flows or patterns fall short

**Design principles applied**
- (e.g. Progressive disclosure, transparency, trust-first, error prevention, calm UX)
- Why these principles are appropriate in this context

**Why this approach (vs alternatives)**
- What trade-off this approach intentionally makes
- What it optimizes for (speed, clarity, confidence, reliability, etc.)

---

## 3. Goal (Why)

### Primary Goal
- What must be true for the user after this flow?
- What success *feels like* from the user’s perspective

### Secondary Goals (Optional)
- Behavioral nudges (confidence, follow-through, reduced drop-off)
- System trust or habit formation outcomes

### Non-Goals (Important)
- What this flow explicitly does **not** try to solve
- Prevents scope creep during prototyping

---

## 4. Flow (How It Works)

### Entry Points
- Where the user comes from
- What context they already have when entering

### Happy Path
1. User sees / selects …
2. System responds by …
3. User confirms / proceeds …
4. Outcome is clearly shown

### Edge / Alternate Paths
- What happens if:
  - Data is missing
  - User abandons mid-flow
  - User returns later

> Keep this **experience-level**, not system-level.

---

## 5. UI Intents

> Describe **what each screen is trying to communicate**, not how it looks.

For each key screen or state:

### Screen / State Name
**Primary intent**
- What the user should immediately understand

**Secondary intent**
- What reassurance, guidance, or motivation is provided

**Emotional tone**
- (e.g. calm, confident, supportive, neutral, transparent)

**Key information hierarchy**
1. Most important
2. Supporting
3. Optional / deferred

---

## 6. Interaction

### User Actions
- What actions are available
- Which action is primary vs secondary

### System Responses
- Immediate feedback (visual, copy, state change)
- Delayed or passive feedback (status, history, confirmation)

### Micro-interactions (Optional but encouraged)
- Highlight changes
- Confirm completion
- Explain disabled or unavailable actions

> Focus on **cause → effect clarity**.

---

## 7. Dependencies

### UX Dependencies
- Other flows that must exist or be understood
- Prior user knowledge or setup required

### Content Dependencies
- Copy, labels, explanations, or empty states needed
- Any assumptions about language or tone consistency

### Experience Dependencies
- Requires trust already established?
- First-time vs returning user differences

---

## 8. Prototype Checklist

Use this before handing off or reviewing:

- [ ] Core concept is understandable in 10 seconds  
- [ ] User goal is explicit and measurable (qualitatively)  
- [ ] Flow covers at least one edge case  
- [ ] UI intent is clear for every key screen  
- [ ] Interactions explain *why*, not just *what*  
- [ ] Dependencies are visible and acknowledged  

---

## 9. Notes & Open Questions

- Open assumptions
- Risks to validate in usability testing
- Questions for stakeholders or future iterations
