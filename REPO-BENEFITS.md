# Repo Folder Benefits

| Folder             | Purpose                                                                  | Benefit                                                                                 | When & How to Use                                                                                                                                        |
| ------------------ | ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `versions/`        | Independent app builds (v1, v2, …)                                       | Experiment freely without breaking working versions; easy rollback by switching folders | Start here for all dev work. `cd versions/v1 && bun run dev`. Create a new `v2/` when starting a major experiment.                                       |
| `fixtures/`        | Shared JSON test data                                                    | Consistent seed data across all versions; no duplication or drift between environments  | Run `bun run db:seed` from within a version. Add new fixture files here when the data model changes — never inside a version folder.                     |
| `guidelines/`      | Visual design system, copy, tokens, deployment guides                    | Single source of truth for UI consistency and team standards across all versions        | Read before touching UI, copy, or deployment config. Update when a design decision becomes a team standard.                                              |
| `common-language/` | Shared terminology, state machines, implementation patterns              | Aligns dev + design vocabulary; reduces ambiguity in PRs, specs, and handoffs           | Reference when naming things, modeling state, or bridging design ↔ dev. Add a file when a new shared concept emerges.                                    |
| `knowledge/`       | Decision log, pattern library, lessons learned, experiment tracker       | Institutional memory — prevents re-litigating past decisions across iterations          | From the repo root, run `bun run knowledge:during` mid-session, `bun run knowledge:weekly` every week, and `bun run knowledge:after` at the end of a version. Read before making architectural decisions. |
| `artifacts/`       | Generated design artifacts (OOUX maps, IA maps, user flows, scopes)      | Traceable design rationale tied to specific dates and iterations                        | Save all design outputs here using the naming convention (`[CODE][YYMMDD]-[slug].md`). Reference by filename in PRs or planning docs.                    |
| `non-analyzed/`    | Artifacts pending review or analysis                                     | Holds imported/migrated artifacts before they've been reviewed and validated            | Move files here when they arrive raw. Once reviewed and contextualized, promote to `artifacts/`.                                                         |
| `skill-command/`   | AI workflow prompts (brainstorming, hypothesis, tradeoffs, ethics gates) | Reproducible AI-assisted workflows; onboards new contributors without tribal knowledge  | Use as input prompts when running structured design or planning sessions with AI. Run in order (1 → 6) for a full problem-framing cycle.                 |
| `templates/`       | ADR, comparison, retrospective, quality gate, dev journal                | Standardized documentation reduces cognitive overhead per iteration                     | Copy the relevant template into `artifacts/` at the start of a session, rename it per convention, then fill it in.                                       |
| `product-context/` | Product roadmap, use cases, screenshots                                  | Keeps product intent visible alongside the code; no context switching to external tools | Read at the start of any new feature or version to align on scope. Update when roadmap or use cases change.                                              |
| `scripts/`         | Knowledge automation (during / weekly / after commands)                  | Reduces friction for capturing learnings at the right moment in the dev cycle           | Run via `bun run knowledge:during`, `knowledge:weekly`, or `knowledge:after` from the repo root. Do not edit unless changing the capture workflow.       |
